# Auto Parts Demand Forecast & Inventory Planner

Forecasts demand-activity for auto-parts categories and turns that forecast into
actionable inventory decisions (safety stock, reorder point, EOQ) — built on a
real marketplace-listings dataset (auto-parts classifieds, all vehicle types).

## ⚠️ Read this first: what this dataset actually is

This is **not** a sales/inventory transaction log. It's a snapshot of a parts
marketplace's active listings (`applications.csv` + supporting lookup tables).
There is no "units sold" or "stock level" field anywhere in the source data.

Two important findings shaped this project's design:

1. **"Demand" is a proxy.** We use *count of new listings per category per day*
   as a stand-in for market demand/activity — the same logic as using search-volume
   or job-posting-volume as an economic proxy when direct transaction data isn't
   available. This is a real and defensible technique, but it measures *supply-side
   posting activity*, not *confirmed sales*. Framed honestly throughout.

2. **Only 30 days of real history exist.** The raw data claims a `app_register_date`
   range of 2021-06-09 to 2024-10-07, but **99.96% of all 95,588 listings fall inside
   a single ~4-week window: Sept 7 – Oct 6, 2024** (matching the `insert_date` scrape
   window of Oct 8 – Nov 2, 2024). The "2021-2023" dates are ~40 stray rows, not a
   real multi-year history. See `reports/figures/00_data_quality_finding.png`.

   **Consequence**: this project forecasts a **short horizon (7 days)** using recent
   trend + day-of-week patterns. It does **not** attempt annual/monthly seasonality
   (e.g. "winter battery demand spike") — there simply isn't enough data for that,
   and claiming otherwise would be dishonest. A real production version of this
   would need at least 1-2 years of daily data to do that reliably.

## What's still genuinely useful here

- **Demand pattern classification** (ADI/CV² — smooth/intermittent/erratic/lumpy)
  is fully valid at any data volume and tells a parts dealer which categories need
  tight forecasting rigor vs. simple reorder rules.
- **Inventory optimization math** (safety stock, reorder point, EOQ) is standard
  operations-research formulas — valid regardless of forecast horizon, as long as
  the demand and variability inputs are sound.
- **The interactive dashboard** lets a user plug in their own current stock and lead
  time assumptions to get a live, category-by-category reorder recommendation.

## Results

| Model | RMSE (7-day holdout) | MAE |
|---|---|---|
| Naive ("same as yesterday") | 623.0 | 259.4 |
| Seasonal naive ("same weekday last week") | 535.3 | 266.2 |
| **XGBoost (global, lag+calendar features)** | 621.2 | 294.1 |

**Honest take**: with only 30 days of data, XGBoost is roughly on par with simple
baselines — it doesn't dramatically outperform naive forecasting yet. This is expected
and reported transparently rather than cherry-picked. Given a real year+ of data, the
same pipeline (lag features, day-of-week, category interactions) would be expected to
pull meaningfully ahead of naive baselines as it could learn real weekly/seasonal cycles.

```bash
python src/data_prep.py          # builds data/processed/daily_category_panel.csv
python src/eda.py                # generates all figures in reports/figures/
```

## Key Design Decisions

- **Category granularity**: level-2 categories (Engine, Filters, Suspension, etc.) —
  23 total, of which 15 have enough volume (≥100 listings in the 30-day window) to
  model individually; the rest are reported but excluded from ML forecasting as too sparse.
- **Global model, not per-category models**: a single XGBoost model with category as
  a one-hot feature, trained across all categories together. With only ~30 daily
  observations per category, a per-category model would badly overfit; pooling data
  across categories gives the model far more to learn from.
- **Recursive multi-step forecasting**: each day's prediction feeds back in as a lag
  feature for the next day's prediction, since we don't have real future lag values.
- **Georgian-script duplicate categories**: 5 level-2 categories are written in
  Georgian script, apparently duplicate branches of English-named siblings (e.g. a
  Georgian "Engine and components" alongside the English one). Translated for
  readability but NOT merged into their English counterparts (no way to confirm
  true equivalence from the data alone) — flagged as a known data-quality issue.
  Their volume is tiny (1-22 listings each) so this doesn't affect the analysis materially.
- **Inventory formula assumptions**: lead time (14 days), order cost ($25), holding
  cost (20%/year), service level (95%) are all industry-typical defaults, fully
  configurable in the Streamlit app, since the source data has no real cost fields.

## Data Quality Notes / Known Limitations

- `avg_price_usd` for "Disassembled vehicles" is ~$0 in the raw data (likely whole
  vehicles aren't priced the same way as individual parts) — this inflates its EOQ
  calculation; flagged rather than silently "fixed."
- No missing values in the core `applications` table.
- `status_id` includes a value (4) not present in `application_status.csv` — left
  as-is (unmapped), doesn't affect the demand analysis.

## Possible Extensions

- Re-run this pipeline once a full year of real data is available — the exact same
  code supports it, just change `RELIABLE_WINDOW_START/END` in `data_prep.py`.
- Incorporate `compatibility.csv`/`vehicles.csv` to forecast demand by **vehicle
  make/model** in addition to part category (e.g. "which parts are trending for
  Toyota vehicles specifically").
- Replace the listing-volume proxy with real transaction data if/when available,
  and validate how well the proxy actually correlated with real sales.
- Try Croston's method for the categories classified as "Intermittent"/"Lumpy" —
  standard ML regression underperforms on sparse demand.
